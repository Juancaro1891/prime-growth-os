# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

PRIME GROWTH OS (`prime-growth-os`) — an AI-powered marketing/growth dashboard for Latin American SMBs, built with Next.js App Router. The UI and all AI prompts are in Spanish (Latin American Spanish, "es-CO" locale). The product pitches itself as an AI "Director de Marketing" (CMO copilot).

## Commands

```bash
npm run dev      # start dev server (next dev)
npm run build    # production build
npm run start    # run production build
npm run lint     # next lint
```

There is no test suite or test runner configured in this repo.

## Environment variables

Required at runtime (set in `.env.local`, gitignored, not present in the repo):

- `ANTHROPIC_API_KEY` — used directly via raw `fetch` calls to `https://api.anthropic.com/v1/messages` (see Architecture below), not just via the SDK.
- `OPENAI_API_KEY` — used directly via raw `fetch` to OpenAI's Images API (`gpt-image-2-2026-04-21`, see `OPENAI_IMAGE_MODEL` in `app/api/generate-image/route.ts`) sent as `Authorization: Bearer <key>`. Same no-SDK convention as the Anthropic calls. `REPLICATE_API_KEY` is no longer read anywhere in the code (the route used to call Replicate's Flux Pro) — it's still harmless to leave in `.env.local`/Vercel, just unused dead config now.
- `NEXT_PUBLIC_SUPABASE_URL` / `SUPABASE_URL` — Supabase REST endpoint, shared by the `leads`, `meta_accounts`, `business_analysis`, and `campaign_suggestions` tables.
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` / `SUPABASE_ANON_KEY` — Supabase anon key, used for the `leads` table. Each var is read with a `NEXT_PUBLIC_*` fallback to a non-public name (see `app/api/leads/route.ts`).
- `SUPABASE_SERVICE_ROLE_KEY` — Supabase service-role key, read via `lib/supabase-admin.ts` and used **only** server-side (never exported to the client) by `lib/meta.ts` (`meta_accounts`), `lib/business-analysis.ts` (`business_analysis`), and `lib/campaign-suggestions.ts` (`campaign_suggestions`). These tables hold sensitive/user-specific data (live Meta access tokens with `ads_management` scope; AI-generated business analysis and campaign drafts) deliberately kept off the anon key + RLS-less path the `leads` table uses — all three have RLS enabled with no public policies, so only the service-role key (which bypasses RLS) can touch them.
- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` / `CLERK_SECRET_KEY` — Clerk auth. See `proxy.ts` and the Architecture section below.
- `META_APP_ID` / `META_APP_SECRET` / `META_REDIRECT_URI` — Meta (Facebook) Marketing API OAuth app credentials, used in `app/api/meta/*` routes. `META_REDIRECT_URI` must exactly match the OAuth redirect URI configured in the Meta App dashboard (currently `https://prime-growth-os.vercel.app/api/meta/callback`).
- `NEXT_PUBLIC_TEST_MODE` — when `"true"`, shows a "⚡ Saltar imágenes (modo prueba)" button in the campaigns launch wizard's AI-image step (`TEST_MODE` constant in `campaigns/page.tsx`) that jumps straight to the final preview with a placeholder instead of calling `/api/generate-image` — for testing the launch flow without spending OpenAI credits. Set in `.env.local` for local dev only; **deliberately not set in Vercel**, since `NEXT_PUBLIC_*` vars get inlined into the client bundle at build time — if it's absent at build time the button code path still ships but always evaluates to hidden, so there's no way to flip it on in production without a redeploy with the var set.

If a key is missing/invalid, `process.env.X` resolves to `undefined`, which gets fetch-serialized into the auth header as the literal string `"undefined"` — the upstream API then rejects it with a generic auth error (e.g. Anthropic's "invalid x-api-key"). That symptom means a missing/wrong env var, not a code bug — check `.env.local` actually has content before debugging the route logic. `next dev` only reads `.env.local` at startup, so a key added while the dev server is already running needs a restart to take effect.

There's no Supabase schema/migration file in the repo:
- `leads` (columns: `id`, `created_at`, `name`, `email`, `phone`, `source`, `stage`, `value`, `notes`) exists only in the Supabase project itself, accessed via the anon key with (presumably) permissive/no RLS.
- `meta_accounts` (columns: `id`, `user_id` [Clerk user id, unique], `access_token`, `expires_at`, `ad_account_id`, `ad_account_name`, `business_name`, `business_category`, `business_description`, `page_id`, `page_fan_count`, `business_data` [jsonb, raw Graph API payload], `created_at`, `updated_at`) also only exists in the Supabase project — RLS is enabled with no policies, so it's only reachable via the service-role key from `lib/meta.ts`.
- `business_analysis` (columns: `id`, `user_id` [Clerk id, **not** unique — one row per analysis run, not one row per user], `icp`, `buyer_persona`, `swot` [jsonb: `{strengths, weaknesses, opportunities, threats}`], `recommended_strategy`, `generated_at`) — same RLS-locked-to-service-role pattern, written by `lib/business-analysis.ts`. To get the latest analysis for a user, query ordered by `generated_at desc limit 1`.
- `campaign_suggestions` (columns: `id`, `user_id` [Clerk id, not unique], `name`, `objective`, `daily_budget`, `copy`, `audience`, `justification`, `status` [default `'pending'`, becomes `'launched'` once approved], `meta_campaign_id`, `meta_status` [`'ACTIVE'`/`'PAUSED'`, Meta's actual delivery status — deliberately a separate column from `status`, see Architecture], `created_at`) — written by `lib/campaign-suggestions.ts`. `status='pending'` rows are the ones a regenerate wipes; `'launched'` rows are permanent history and never deleted by the app.

## Architecture

**Routing has two parallel trees.** Every top-level route under `app/<name>/page.tsx` (e.g. `app/dashboard`, `app/crm`, `app/analytics`, ...) is just a `redirect()` stub pointing to `app/dashboard-layout/<name>/page.tsx`, where the real page and its sidebar chrome live. This split exists so `app/dashboard-layout/layout.tsx` can wrap all real pages in the shared `<Sidebar />` without affecting the root layout. **When asked to modify a page, edit the file under `app/dashboard-layout/`, not the top-level redirect stub.**

**Pages fall into two categories:**
- *Live/wired*: `dashboard` (chat), `analyzer`, `creatives`, `strategy`, `images`, `crm`, `campaigns` call real API routes (`/api/chat`, `/api/analyze-business`, `/api/generate-copy`, `/api/generate-strategy`, `/api/generate-image`, `/api/leads`, `/api/meta/*`).
- *Static mocks*: `analytics`, `automation`, `landings`, `playbooks`, `whatsapp` render hardcoded in-file sample data (gym-business scenario) with no backend calls. Don't assume there's a hidden API behind their numbers — there isn't.

**Two standalone public pages bypass the dashboard tree entirely**: `app/landing/page.tsx` (marketing/sales page) and `app/lead-capture/page.tsx` (generic embeddable lead form). Both live directly under `app/`, not under `app/dashboard-layout/`, so they render through the bare root layout with no `<Sidebar/>` — each has its own `layout.tsx` sibling only to set page-specific `<title>`/meta.

**Auth is Clerk, enforced in `proxy.ts` (repo root, not `app/proxy.ts` — Next.js only picks up middleware/proxy at the project root). Next.js 16 renamed the `middleware.ts` file convention to `proxy.ts`** (same behavior, file/export renamed); don't recreate a `middleware.ts` here, it'd just trigger the deprecation warning again. `clerkMiddleware` + `createRouteMatcher(["/dashboard-layout(.*)"])` calls `auth.protect()` only for `/dashboard-layout/*`; everything else (`/landing`, `/lead-capture`, `/api/leads`, `/sign-in`, `/sign-up`, and all other `/api/*` routes) is reachable without a session at the middleware level. Individual `/api/*` route handlers that need a real user (currently `app/api/meta/*`) call `auth()` themselves and return 401 — the middleware running on `/api/*` only attaches Clerk's auth context, it doesn't block anything there. `app/layout.tsx` wraps the app in `<ClerkProvider>`. Custom-styled sign-in/up pages live at `app/sign-in/[[...sign-in]]/page.tsx` and `app/sign-up/[[...sign-up]]/page.tsx` (catch-all routes, required by Clerk's `<SignIn>`/`<SignUp>` components for multi-step flows), sharing one `appearance` theme object from `lib/clerk-appearance.ts`. `NEXT_PUBLIC_CLERK_SIGN_IN_URL=/sign-in` / `NEXT_PUBLIC_CLERK_SIGN_UP_URL=/sign-up` are required in env — without them `auth.protect()` redirects to Clerk's hosted Account Portal instead of these pages.

**AI calls bypass the Anthropic SDK.** All `app/api/*/route.ts` handlers call `https://api.anthropic.com/v1/messages` directly via `fetch` (commit "Fix API routes - use fetch instead of SDK"), each with its own inline `SYSTEM_PROMPT`/prompt string. `lib/anthropic.ts` exports an `Anthropic` SDK client and a `SYSTEM_PROMPT` but **is dead code — nothing imports it**. The canonical, currently-used system prompt for the chat copilot lives in `app/api/chat/route.ts`, not in `lib/anthropic.ts`; keep both in sync (or remove the unused one) if you change the assistant's persona/instructions.

**Supabase access is hand-rolled REST, no SDK.** `app/api/leads/route.ts` talks to Supabase's PostgREST endpoint (`${url}/rest/v1/leads`) directly via `fetch` with `apikey`/`Authorization` headers — there is no `@supabase/supabase-js` dependency. CRM UI (`app/dashboard-layout/crm/page.tsx`) calls this route for list/create/patch (stage changes via Kanban or detail panel).

**JSON-from-LLM parsing pattern**: `analyze-business`, `generate-copy`, and `generate-strategy` routes prompt Claude to return raw JSON, strip ` ```json ` fences with a regex, then `JSON.parse` the result. There's no schema validation beyond that — if you add a new structured-output route, follow the same strip-then-parse convention used there. `generate-strategy` additionally falls back to extracting a `{...}` substring via regex if the direct parse fails, and logs the raw model output on any failure — worth copying into new routes with large/nested JSON schemas, since bigger schemas are more likely to get truncated by `max_tokens` mid-response.

**`generate-image` doesn't need the strip-fence dance** — it calls OpenAI's Images API with `gpt-image-2-2026-04-21` (`OPENAI_IMAGE_MODEL` constant in `app/api/generate-image/route.ts`; previously `gpt-image-1`, swapped models with no other code changes since the parameter shapes that matter here carried over), which returns structured JSON natively, no markdown-wrapped text to parse, and (unlike DALL-E 3 before it, and unlike the Flux Pro/Replicate version that briefly replaced DALL-E 3) is fully synchronous — no async prediction/polling loop. **gpt-image-2 has no `response_format`/url option — it always returns base64 (`data[0].b64_json`)**, same as gpt-image-1 — confirmed with a real low-quality API call against both `/v1/images/generations` and `/v1/images/edits`, not assumed from docs (OpenAI's `/v1/images/edits` reference page doesn't list `gpt-image-2` as supported at all as of this writing, but it works in practice — docs lagging a 2-month-old model). The route wraps the base64 as a `data:image/png;base64,...` URI and returns it in the same `{ url }` field the frontend has always expected. **Sizes**: gpt-image-2 supports arbitrary resolutions (multiples of 16px, max edge 3840px, aspect ratio ≤3:1, 655,360–8,294,400 total px) instead of gpt-image-1's fixed enum (`1024x1024`/`1536x1024`/`1024x1536`/`auto`) — but the three concrete values already in use satisfy gpt-image-2's broader constraints too, so `openAiSizeBySize` didn't need to change. The frontend still sends DALL-E-3-shaped size strings (`images/page.tsx`'s format picker, the wizard's hardcoded `"1024x1024"`), so `openAiSizeBySize` in the route remaps them (`1792x1024→1536x1024`, `1024x1792→1024x1536`) — if you add a new size option in either frontend, add the mapping here too or it'll silently fall back to `1024x1024`. Verify any future model swap against the live API directly (`GET /v1/models/{id}` plus a real low-quality test call), not just docs — both have lagged/been wrong for brand-new model releases in this exact file's history.

**gpt-image-2 at `quality: "high"` is slow — ~168s observed for one `/v1/images/generations` call**, vs. ~38–49s for gpt-image-1 at the same quality. `export const maxDuration` was bumped from `60` to `300` because of this — `next dev` ignores `maxDuration` entirely (so this never showed up locally), but on Vercel a request running past the configured duration gets killed by the platform before OpenAI's response arrives, the client gets no JSON body at all, and the wizard's `firstError` extraction (`r.data?.details || r.data?.error`) has nothing to read, silently falling through to its generic "No se pudo generar ninguna imagen" string — that symptom looked like a missing-error-message bug but was actually a platform-level timeout. If `maxDuration: 300` still isn't enough in production, it's a Vercel plan ceiling (verify the current numbers at vercel.com/docs/functions/configuring-functions/duration, they've changed over time — as of mid-2026, with Fluid Compute, Pro's default is already 300s and its non-extended max is 800s), not a code bug — check the plan before raising this number further, or drop `quality` to `"medium"`/`"low"` instead.

**The project's Vercel plan is Pro.** With Fluid Compute (on by default), **the platform's own default `maxDuration` when a route doesn't set one at all is already 300s** on Hobby/Pro/Enterprise alike — it is *not* the old 10–15s default from pre-Fluid-Compute Vercel, so most routes here (`chat`, `analyze-business`, `generate-copy`, `generate-strategy`, `leads`, `meta/auth`, `meta/campaigns`, `meta/create-campaign`) need no explicit `maxDuration` at all; they already get 300s. `generate-image`, `meta/callback`, `meta/generate-campaigns`, and `meta/adjust-suggestion` set `maxDuration = 300` explicitly anyway — for `generate-image` because 300 is the empirically-needed value (see above), and for the other three only because they used to hardcode `60`, which is *more* restrictive than the platform default and was tightening a budget that didn't need tightening (those three call Claude, and `meta/callback`'s `after()` background job additionally does Graph API calls — none of that is known to need more than 300s, but there was no reason to cap it below the default either). If you add a new route that does a slow external call (LLM, image gen, etc.), check whether you actually need to override `maxDuration` before adding one — the unset default is probably already enough. `generate-image/route.ts` also retries once against `gpt-image-1` (`OPENAI_IMAGE_FALLBACK_MODEL`) specifically when `isModelUnavailableError()` matches the failure (HTTP 404, `code: "model_not_found"`, or message text like "does not exist"/"do not have access") — this fallback does **not** trigger on timeouts or other error types, only on model-availability errors, by design (don't widen the heuristic to "any failure" without thinking about what that masks).

**Image-to-image / reference-image support moved from Flux Redux's `image_prompt` field to OpenAI's separate `/v1/images/edits` endpoint**, which is multipart/form-data, not JSON — a fundamentally different request shape from `/v1/images/generations`. `generate-image/route.ts` branches on whether the request body's `imagePrompt` (still the same field name the frontend already sends, a base64 data URI) is present: if so, it decodes the data URI back into a `Blob` (`dataUrlToBlob()`, via `Buffer.from(..., "base64")`) and posts it as `image[]` to `/v1/images/edits` instead of `/v1/images/generations`. `app/dashboard-layout/campaigns/page.tsx`'s `LaunchWizardModal` is the only current caller that ever sets `imagePrompt` (see below); `images/page.tsx` doesn't, so it always hits `/v1/images/generations`. Conceptually this is a real semantic shift from the old Flux Redux behavior — `/v1/images/edits` literally edits the supplied image per the prompt rather than generating a fresh image loosely inspired by it — but the wiring (one optional reference image, used as-is) is unchanged.

**Meta (Facebook) Ads integration lives under `app/api/meta/*`, sharing helpers from `lib/meta.ts`.** OAuth flow: `GET /api/meta/auth` requires a Clerk session, generates a random `state`, stores it in an httpOnly `meta_oauth_state` cookie, and 302s to Facebook's OAuth dialog with `scope=ads_management,ads_read,business_management,pages_show_list,pages_read_engagement` (the `pages_*` scopes exist solely so `/me/accounts` returns page data — see below; without them the auto-enrichment step silently gets an empty page list). `GET /api/meta/callback` (the configured `META_REDIRECT_URI`) verifies `state` against that cookie (CSRF check), exchanges `code` for a short-lived token, exchanges that for a long-lived (~60 day) token, fetches the user's first ad account via `getAdAccounts()` (`/me/adaccounts`, fields `id,name,currency,timezone_name`), and upserts `{ user_id, access_token, expires_at, ad_account_id, ad_account_name }` into `meta_accounts` keyed on `user_id` (Clerk id) — then immediately redirects to `/dashboard-layout/dashboard?meta_connected=true` (or `?meta_error=<reason>`). `GET /api/meta/campaigns` and `POST /api/meta/create-campaign` both call `getMetaAccount(userId)` to look up the stored token/ad account and proxy straight to the Marketing API (`${ad_account_id}/campaigns`); there's no per-request `adAccountId` param — it's always whichever ad account was first returned by `/me/adaccounts` at connect time. `create-campaign` defaults `status` to `PAUSED` if the caller doesn't specify one, so wiring up the route can't accidentally start real ad spend. `create-campaign` also accepts an optional `suggestionId`; if present (and owned by the calling user), it marks that `campaign_suggestions` row `status: "launched"` with the new `meta_campaign_id` after a successful create — see below. `GET /api/meta/campaigns` has no frontend page yet; the other three are wired into `app/dashboard-layout/campaigns/page.tsx`.

**After the redirect, the callback schedules a background job via `after()` from `next/server`** (`enrichAndAnalyze()`, defined inline in `app/api/meta/callback/route.ts`, `export const maxDuration = 60`) — this is "Step 1 of the autonomous marketing agent." It fetches the user's Facebook Pages via `getFacebookPages()` (`/me/accounts`, fields expanded to include `category,description,about,fan_count,website` directly — no second per-page request needed), takes the first page, and re-upserts `meta_accounts` with `business_name`/`business_category`/`business_description`/`page_id`/`page_fan_count`/`business_data` (raw pages+ad-account payload) filled in. It then calls `generateBusinessAnalysis()` from `lib/business-analysis.ts` — a Claude call following the same strip-fence/regex-fallback JSON pattern as `generate-strategy` — to produce `{ icp, buyerPersona, swot, recommendedStrategy }` from whatever business data was available, and `saveBusinessAnalysis()` inserts that into `business_analysis`. This whole job is deliberately decoupled from the OAuth redirect: any failure in it (missing scopes, Claude error, no pages found) is caught and logged, never surfaces to the user, and never undoes the already-saved Meta connection — `enrichAndAnalyze`'s job is enrichment, not the connection itself.

**Step 3 of the autonomous agent — `app/dashboard-layout/campaigns/page.tsx` plus `lib/campaign-suggestions.ts`, `app/api/meta/generate-campaigns/route.ts`, `app/api/meta/adjust-suggestion/route.ts` — turns the saved `business_analysis` into launchable Meta campaigns.** The page has three states driven by one `GET /api/meta/generate-campaigns` call (`{ connected, businessName, hasAnalysis, suggestions }`): not connected → "Conectar Meta Ads" button (plain navigation to `/api/meta/auth`, not a fetch, since it has to be a top-level browser navigation into Facebook's OAuth dialog); connected with no suggestions at all → "Generar mi estrategia de campañas con IA" button; connected with any suggestions → a card grid. **`GET` returns every suggestion regardless of `status` (pending *and* launched)** — it used to filter to `status='pending'` only, which made launched campaigns (and their activate/pause controls, see below) silently vanish from the page on every reload; only the `POST` (regenerate) path still filters to `status='pending'` when deciding what to delete/replace. `POST /api/meta/generate-campaigns` reads the latest `business_analysis` row (`getLatestBusinessAnalysis`, gracefully `null` if Step 1's background job hasn't finished or failed — the prompt just tells Claude to infer from category/name instead) plus `meta_accounts.business_data.adAccount` (currency/timezone, nested in the jsonb blob from Step 1, not its own column) and asks Claude for exactly 3 campaigns, one per objective (`OUTCOME_TRAFFIC`/`OUTCOME_LEADS`/`OUTCOME_AWARENESS`, whitelisted server-side in `normalizeObjective()` — anything else from Claude silently falls back to `OUTCOME_LEADS`). `replacePendingSuggestions()` deletes that user's existing `status='pending'` rows and inserts the new 3 — `launched` rows are never touched, so "Regenerar todo" can't erase campaigns that were already approved. "Ajustar con IA" on a card posts free-text `instruction` to `/api/meta/adjust-suggestion`, which re-prompts Claude with the current suggestion + instruction and overwrites that one row in place (ownership is re-checked via `getCampaignSuggestion(id, userId)` before any read or write — ids are UUIDs but the check is defense-in-depth, not obscurity). "Aprobar y lanzar" no longer calls `create-campaign` directly — it opens `LaunchWizardModal` (defined inline in `campaigns/page.tsx`, per the "no new files" constraint of that change), a 3-step creative wizard: step 1 picks AI-generated vs. user-uploaded images; step 2 either generates images or collects up to 3 drag-and-drop uploads (Feed/Stories/Banner) as local blob URLs via `URL.createObjectURL` — uploaded/generated images are **not** sent to any Meta endpoint; there's no AdImage/AdCreative/Ad-level integration yet, only the bare Campaign object via `create-campaign`, so the picked image is purely a UI thumbnail for step 3's confirmation screen. Step 3 still calls `create-campaign` with `suggestionId`, which is what actually flips that row to `launched` (and sets its initial `meta_status` to whatever was actually sent to Meta — `"PAUSED"` by default). None of this validates that `daily_budget` is in the ad account's minor currency unit (e.g. cents for USD) before forwarding it as `daily_budget` to the Marketing API — same simplification the route already had.

**Activating/pausing a launched campaign is real now, via `app/api/meta/toggle-campaign/route.ts` and `lib/meta.ts#updateCampaignStatus()`.** `campaign_suggestions.meta_status` (`"ACTIVE"`/`"PAUSED"`) is a **separate column from `status`** (`"pending"`/`"launched"`) on purpose — `status` is the internal workflow stage that `SuggestionCard`'s `launched` check and the launch wizard depend on; overloading it with Meta's delivery state would break "is this already launched" logic the moment someone paused a campaign. `toggle-campaign` takes `{ campaignId, action }`, verifies `campaignId` (Meta's numeric id, i.e. `meta_campaign_id`) belongs to the calling user via `getCampaignSuggestionByMetaCampaignId()` before calling Meta — Meta's own API would also reject a token that doesn't administer that campaign, this check is defense-in-depth on top of that, same pattern as the other `campaign_suggestions` routes. On success it writes `meta_status` back via the same `updateCampaignSuggestion()` used everywhere else. `SuggestionCard` shows a 🟢/⏸️ badge + a single toggle button instead of the launch/adjust buttons once `status==='launched' && meta_campaign_id` is set; the wizard's success screen offers "▶ Activar ahora" (calls `toggle-campaign` with `action:"ACTIVE"` before closing) or "Activar después" (closes with `meta_status:"PAUSED"`, no API call). **Toggling status here does not actually start real ad spend** — `create-campaign` only ever creates the bare Campaign object, with no AdSet (budget) or Ad (creative) beneath it, and Meta requires both before a campaign delivers anything regardless of its `status`. So "Activar ahora" is currently safe to expose with zero extra confirmation, but the UI copy ("para empezar a recibir resultados") oversells what flipping this flag actually does today — it'll start meaning that the moment AdSet/Ad creation gets built on top of this.

The AI-image branch of step 2 (`aiPhase` state: `"customize"` → `"generate"`) opens on an optional "Personaliza tus creativos" screen first: up to 3 reference-photo dropzones (`referenceSlots`, generic — no fixed dimensions, unlike the upload path's Feed/Stories/Banner) plus one logo dropzone, both skippable via a single button that reads "Continuar sin fotos" or "Continuar con estas fotos" depending on whether anything was uploaded. Only the **first** uploaded reference photo is actually used (if multiple are uploaded, the rest are silently ignored) — both it and the logo are converted client-side to base64 data URIs via `fileToDataUrl()` (`FileReader.readAsDataURL`) and sent as separate fields, `imagePrompt` and `logoPrompt`, to both parallel `/api/generate-image` calls.

**The logo's actual pixels ARE sent to OpenAI now** — confirmed against the live API, not assumed from docs: `/v1/images/edits` accepts up to 16 images via repeated `image[]` form parts (verified with a real 2-image call against `gpt-image-2-2026-04-21`: gave it a fake "reference photo" + a fake "logo" and a prompt saying "the first image is X, the second is Y, do Z" — the model correctly placed the second image's content per the instructions referencing it by position, not by filename or any metadata). `generate-image/route.ts` builds the `image[]` array in a fixed order — `image[0]` = reference photo (only if `imagePrompt` present), then logo (only if `logoPrompt` present, becoming `image[0]` on its own if there's no reference photo) — and the wrapping prompt text changes based on which of the two are actually present, because they need contradictory instructions: the reference photo is loose style/color/composition guidance ("don't copy literally, generate something new inspired by it"), while the logo must be reproduced exactly ("respect its exact colors/typography/shape") — mixing those into one generic sentence when both are attached would tell the model contradictory things about the same kind of input. `callOpenAiImages()` takes an `images: ImageBlob[]` array (not a single optional image anymore) and only calls `/v1/images/edits` when that array is non-empty; with zero images it calls `/v1/images/generations` same as before. `buildImagePrompt(suggestion, hasLogo)`'s logo branch text now says the logo "se proporciona como imagen de referencia" — that phrasing is no longer aspirational, it's literally true. When `hasLogo` is false the prompt still says "sin logos ni marcas visibles" — keep that `else` branch, since dropping it would make a logo-less generation share wording with a logo generation, exactly backwards.

**The image prompt is user-editable, not just generated and sent as-is.** `imagePromptText` state holds whatever text is actually POSTed to `/api/generate-image`; it's seeded from `buildImagePrompt(suggestion, hasLogo)` the moment "Continuar sin/con fotos" advances `aiPhase` to `"generate"`, then rendered as a plain `<textarea>` the user can edit before the first "Generar imágenes" click. After images come back, the textarea is hidden by default but can be re-expanded via a "✏️ Ajustar prompt" toggle (`showPromptEditor`) without losing the already-generated images or resetting `aiPhase`/`step` — editing it there and clicking the existing "Regenerar" button (not a separate action) picks up the edited text, since both buttons call the same `handleGenerateImages()`, which always reads current `imagePromptText`. Note `handleGenerateImages` clears `aiImages` synchronously at the top, so mid-regeneration the view always falls back to the pre-generation render branch (textarea + button) regardless of which button triggered it. A 5-second-interval rotating status message (`imageLoadingMessages`, cycled by `loadingMessageIndex` in a `useEffect` keyed on `aiGenerating`) replaces a static "esperá 1-3 minutos" notice once generation actually starts — the static notice and the rotating one are mutually exclusive on `aiGenerating`, not both shown at once.

**Lead `source` values are an evolving, ungoverned string enum.** `app/dashboard-layout/crm/page.tsx` keeps a local `sourceColors` map and a hardcoded option list for the "new lead" modal; anything posting to `/api/leads` from outside the CRM (the `/landing` and `/lead-capture` public forms) invents its own `source` string ("Landing", "Formulario Web"). When adding a new lead-capturing surface, add its source string to both `sourceColors` and the select options in `crm/page.tsx` too, or it'll just render with the gray fallback badge.

**Chat streaming**: `app/api/chat/route.ts` proxies Anthropic's SSE stream, parsing `content_block_delta` events and re-emitting just the text deltas as a plain chunked `text/plain` stream (not SSE) — the client (`app/dashboard-layout/dashboard/page.tsx`) reads it with a raw `ReadableStream` reader, not an SSE/event-source client.

**Styling**: Tailwind only, dark theme hardcoded (`bg-[#0a0a0f]`, violet/purple accents). No CSS modules or styled-components. `lib/utils.ts#cn()` (clsx + tailwind-merge) is the standard way to compose conditional class names — used throughout `components/`.

**Path alias**: `@/*` maps to the repo root (see `tsconfig.json`), e.g. `@/lib/utils`, `@/components/sidebar`.
